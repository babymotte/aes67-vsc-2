High level receiver buffer API:
- one buffer per receiver is created for specific audio format (sample rate, bit depth, channel number)
- buffer internally stores f32 data
- buffer is a ring buffer, writing data to the buffer overwrites old data
- buffer must be large enough that it can contain the link offset multiple times -> what happens if link offset changes?
- writing to the buffer must never block
- reading from the buffer can be done in a blocking way (wait for data to be available) or non-blocking (method reports number of written samples per channel)
- buffer internally deinterlaces channels so data can be queried for individual single channels

// IMPORTANT:
// send buffer is JACK/ALSA SPECIFIC. It must be re-usable but it must NOT be used in lower level applications like a DSP
// the general sender API MUST NOT rely on it! To the sender API the buffer must act like a regular producer
High level sender buffer API:
- one buffer per sender is created for specific audio format (sample rate, bit depth, channel number)
- buffer internally stores f32 data
- buffer is a ring buffer, writing data to the buffer overwrites old data
- buffer must be large enough that it can contain the send offset multiple times -> what happens if send offset changes?
- egress time is the time of samples leaving the buffer, not the underlying producer!
- writing to the buffer never blocks
- writing to the buffer triggers push of data to sender, which may block
- data takes and stores deinterlaced data, does the sender accept deinterlaced or only interlaced data?
